# AI Developer Skillset Implementation Plan

Goal: Improve the dev-kit skillset by synthesizing the upstream reference skills into a clearer, more complete workflow for developers using AI agents.
User outcome: Installed agents route to the right workflow more reliably, preserve offline behavior, and give developers stronger guardrails for product thinking, implementation, verification, review, and skill authoring.
Architecture: The canonical source remains `skills/`; provider-specific files are generated by `scripts/install.sh` and `scripts/install.ps1`. New or revised skills must be reflected in `skills/registry.json`, README routing, installer routing blocks, and structure/install tests.
Risk: Feature
Execution: Use `execute-work` for task-by-task implementation.
Verification: Use `verify-work` before any completion claim.

## Source Evidence

- Current installed workflow: `start-work -> plan-work -> execute-work -> verify-work -> capture-learning -> finish-work` in `README.md` and `AGENTS.md`.
- Canonical skills live under `skills/`; `.opencode/skills/` is an installed workspace copy.
- Upstream Superpowers contributes: `brainstorming`, `using-git-worktrees`, `writing-plans`, `subagent-driven-development`, `executing-plans`, `dispatching-parallel-agents`, `test-driven-development`, `requesting-code-review`, `receiving-code-review`, `finishing-a-development-branch`, `systematic-debugging`, `verification-before-completion`, and `writing-skills`.
- Upstream AI DevKit contributes: `dev-lifecycle`, `agent-orchestration`, `debug`, `verify`, `memory`, `capture-knowledge`, `technical-writer`, `simplify-implementation`, and `tdd`.
- Existing CI commands are `bash -n scripts/install.sh`, `bash tests/test-skill-structure.sh`, `bash tests/test-install.sh`, and `bash tests/test-upstream-references.sh`.
- The dev-kit must remain offline-installable: no `npm`, `npx`, network fetches, or external CLIs in normal workflows.

## Proposed Skillset Shape

Keep the current small core, but add two missing workflow-level skills where the upstream references show durable value that is not yet first-class:

- `shape-work`: product and technical shaping before implementation planning. Combines Superpowers `brainstorming` with AI DevKit requirement/design review. Use for vague ideas, product direction, design alternatives, or when requirements are not concrete enough for `plan-work`.
- `review-work`: implementation review before finishing. Combines Superpowers `requesting-code-review` with AI DevKit `code-review` and the existing spec-before-quality discipline. Use after a task, before merge/PR, or when the user asks for code review. This keeps review distinct from `finish-work`, which handles handoff, commit, and PR readiness.

Revise existing skills lightly instead of adding more workflow names:

- `start-work`: route vague product ideas to `shape-work`; route concrete multi-step implementation to `plan-work`; keep tiny fast path.
- `plan-work`: accept shaped briefs as input; require acceptance criteria and verification commands.
- `execute-work`: reference `review-work` as the review gate when not using subagents.
- `finish-work`: call `review-work` before final handoff when no review evidence exists.
- `writing-skills`: add a small eval-output requirement for changed skills, based on upstream skill evaluation guidance.

Do not add a separate `worktree` skill yet. `start-work` already covers isolation, and adding another skill would increase routing complexity without enough user-facing benefit in this repo.

# Shape Work

## Purpose
Turn an idea into implementable requirements without coding.

## When To Use
- Vague product idea.
- Competing approaches.
- User asks to brainstorm, shape, explore, or decide what to build.
- `start-work` finds requirements are not concrete enough for `plan-work`.

## When Not To Use
- Bug reports: use `debug-root-cause`.
- Written requirements already exist: use `plan-work`.
- A plan is ready: use `execute-work`.
- Tiny mechanical edits: use direct execution plus `verify-work`.

## Hard Rules
- Do not write implementation code.
- Do not invent user requirements when evidence is missing.
- Ask only questions that repo docs, memory, and code cannot answer.
- Prefer the smallest coherent product slice.
- Keep dev-kit workflows offline-safe.

## Workflow
1. Read the start brief and relevant docs.
2. Identify user, job-to-be-done, and current pain.
3. List constraints from repo, user instructions, offline requirements, and risk.
4. Compare 2-3 approaches.
5. Pick the smallest valuable slice and explicitly reject out-of-scope ideas.
6. Define acceptance signals and verification ideas.
7. Produce a shaping brief and recommend `plan-work` when requirements are ready.

# Review Work

## Purpose
Find real implementation risks before handoff.

## When To Use
- User asks for review.
- After implementation task before marking complete.
- Before `finish-work` when no review evidence exists.
- Reviewing a diff against a plan or acceptance criteria.

## When Not To Use
- Bug diagnosis: use `debug-root-cause`.
- Planning: use `plan-work`.
- Product shaping: use `shape-work`.
- Skill authoring review: use `writing-skills` plus this only for diff risks.

## Hard Rules
- Findings first, ordered by severity.
- Do not praise before findings.
- Verify claims against the diff, plan, and tests.
- Spec compliance review comes before code quality review.
- Critical issues block completion.

## Workflow
1. Identify review target: diff, files, plan, or branch.
2. Read the relevant plan/requirements when present.
3. Inspect changed files and tests.
4. Pass 1: spec compliance.
5. Pass 2: correctness, security, edge cases, maintainability, test quality.
6. Report findings with severity and file/line references.
7. If no findings, state residual risks and verification gaps.
